# Tirion — Stock Fulfillment View Spec

## Purpose

The Stock Fulfillment View is a planning-phase workspace where planners
identify Work Orders whose demand can be satisfied from existing stock, and
either pull from stock to fulfill them or pass them through to procurement.

This view is the second step in the operational sequence. It receives WOs
generated by Project Creation in `Unreleased` state. The planner records
decisions on each candidate (Fulfill from Stock or Pass Through), and then
explicitly releases the decided Pass-Through WOs into operational visibility
via the Release button. Released WOs become visible to Batching Lens,
Operations Lens, Project View, and execution lenses.

The view is a workspace, not an automatic flow. The planner controls when
WOs leave Stock Fulfillment for downstream views. This allows the planner
to hold back specific WOs that need additional consideration without
blocking progress on the rest.

The view is intentionally narrow in scope: it shows only WOs that are
*candidates* for fulfillment (Stock Count ≥ Demand). All other WOs in active
Projects are pre-determined to require procurement — they are still tracked
in the system as decided non-candidates, and they release alongside the
candidates when the planner clicks Release.

This view is also the system's primary opportunity for passive inventory
reconciliation. Because the planner is physically interacting with stock
during fulfillment, discrepancies between recorded and actual counts surface
naturally. The view supports inline reconciliation as a free side-effect of
the workflow — no separate trip to Parts Master required.

## Operational Sequence

```
Project Creation → Stock Fulfillment → [Release] → Batching Lens → Execution Lenses
                              │
                              └── Fulfill from Stock → Complete (immediate operational visibility)
```

A Project compiled by Project Creation has all its WOs in `Unreleased` state
with `stockFulfillmentReviewedAt = null`. Those WOs that qualify as
fulfillment candidates (Stock ≥ Demand) appear in this view. The planner
works through candidates and records decisions. Non-candidates do not
appear in the view but are tracked as "decision pending."

**Two paths out of Unreleased:**

- **Fulfill from Stock:** WO transitions `Unreleased → Complete` immediately
  via Skipped steps, with stock decrement. No release event needed — the WO
  is done. It is immediately visible to Operations Lens, Project View, and
  any other view that shows WOs in non-Unreleased states
- **Pass Through (explicit, auto, or non-candidate at Release):** WO retains
  `Unreleased` state with `stockFulfillmentReviewedAt` stamped. Stays
  Unreleased until the planner clicks Release. On Release, transitions
  `Unreleased → Open` and becomes visible to Batching, Operations Lens,
  execution lenses, etc.

This is a per-WO model, not a per-Project gate. Projects do not "clear"
Stock Fulfillment — individual WOs do. New WOs added to Active Projects
(via Add Top-Level Item in Project Creation View) appear in Stock
Fulfillment as new candidates, joining the workspace alongside any held-back
WOs from prior sessions.

## Entry Conditions

A Work Order appears as a candidate in this view when ALL of the following
are true:

1. WO is in `Unreleased` state
2. The Part referenced by the WO has `stockCount ≥ WO.quantity`
3. The WO's parent Project has `status = Active`
4. The WO has not already been marked reviewed (`stockFulfillmentReviewedAt
   = null`)
5. The WO is not Cancelled (Cancelled WOs are terminal and excluded from
   all operational visibility)

A Work Order disappears from candidacy mid-session when:

1. Stock Count for its Part drops below its Demand (e.g., another candidate
   consumed available stock). When this happens, the WO has its
   `stockFulfillmentReviewedAt` stamped and disappears from the candidate
   list — but it remains in `Unreleased` state until the planner clicks
   Release
2. The WO is fulfilled (transitions to Complete) or explicitly passed
   through by the planner
3. An ancestor Assembly is fulfilled, cascading Skip-Complete down the tree
   (the descendant WOs become Complete and exit the view)
4. The WO is Cancelled mid-session (via the Cancel primitive applied
   elsewhere). The WO is silently removed from the candidate list

**Hard rule:** The view never shows a WO that cannot be fully satisfied from
current stock. Partial fulfillment is not supported. The system pre-filters
the candidate list — no fractional or insufficient-stock decisions are
surfaced to the planner.

**Hard rule:** Decisions and Release are separate events for Pass-Through
WOs. A Pass-Through WO has its decision marker (`stockFulfillmentReviewedAt`)
set immediately, but does not transition state until the planner clicks
Release. Fulfilled WOs do not require a release step — they transition
to Complete at fulfillment time.

**Hard rule:** Cancelled WOs do not appear in this view under any
circumstance, including via filters. Cancellation removes a WO from
operational scope entirely.

## Defensive Validation

Validation issues that should have been caught at Project Creation are
defensively re-checked here, but they do not gate the view:

- A WO with no Routing Template should be impossible (Project Creation gate
  prevents it). If it occurs, the WO would only appear here if stock is
  sufficient, and routing has no bearing on Fulfill from Stock — the action
  works regardless. No special handling required
- A WO for an inactive Part should be impossible (same gate). Defensive
  re-check with inline indicator, no special handling
- A WO with no MaterialSpec is normal and unremarkable — not an error state

These defensive considerations exist for forward-compatibility (e.g., if
project-edit features are added in a future revision and bypass the
Project Creation validation gate, this view should remain robust).

---

## Defensive Rule — Descendant-Then-Ancestor

When a descendant WO is fulfilled (or otherwise transitions to Complete),
any ancestor Assembly WOs in the same Project that were candidates lose
their candidate status. Pulling the ancestor would attempt to cascade-skip
WOs that are already Complete via real fulfillment, which is structurally
invalid and operationally wasteful (the ancestor's pull would duplicate
parts already physically allocated).

The system enforces this with two layers:

**Pre-filter (UX layer):** When a descendant WO transitions to Complete,
the candidate list re-evaluates eligibility. Any ancestor Assembly WO in
the same Project drops out of the candidate list immediately. The planner
never sees the option to pull the ancestor.

**Action-time check (structural backstop):** When a Fulfill action targets
an Assembly, the system checks every descendant WO in the same Project. If
any descendant is already Complete, the action is blocked with a message
explaining why ("Cannot fulfill [Assembly Y] from stock — [Part X] has
already been fulfilled separately. Pulling this Assembly would duplicate
the fulfillment.").

The pre-filter handles normal usage cleanly. The action-time check
defends against any race condition or edge case the pre-filter might miss.

**Hard rule:** An Assembly WO loses candidate status the moment any of its
descendant WOs in the same Project transition to Complete via any path.

**State of de-listed ancestors:** The Assembly WO whose candidacy was
revoked is not auto-passed-through immediately. It remains Unreleased and
becomes a non-candidate for the rest of the planner's session. When the
Project clears the gate, it gets the standard non-candidate marker.

The inverse case — descendant becomes a non-candidate when its ancestor
is fulfilled — is handled by the standard cascade behavior. The ancestor's
fulfillment cascades the descendant to Complete, which removes it from the
candidate list cleanly. No additional defensive logic needed.

## View Structure

### Project Header Section

The view is organized around Project context. Each Project with at least one
candidate or one decided-but-unreleased WO appears as a header in the view's
top section.

Each Project header shows:

| Element | Notes |
|---------|-------|
| Project Number + Top-Level Reference(s) | e.g., "20137 — 20137.01, 20137.02" |
| Project Name | |
| Color Indicator | Project color (system-wide visual convention — Project records receive a color at creation; consistent across all views) |
| Candidates Remaining | Number of WOs in this Project still candidates (no decision recorded) |
| Decided but Unreleased | Number of WOs in this Project with a decision recorded but not yet released |
| Release Button | "Release Project [Number] — N WOs" — releases this Project's decided-but-unreleased WOs only |

Clicking a Project header focuses the candidate list on that Project. A "Show
all Projects" toggle returns to the unified view.

A Project's header disappears from the view when:
- All candidates have decisions recorded AND
- All decided WOs have been released

A Project's header reappears if new WOs are added (e.g., via Add Top-Level
Item in Project Creation View) and they qualify as candidates or non-candidate
review-pending.

### Global Release Button

Above the Project header section, a global Release button releases all
decided-but-unreleased WOs across all visible Projects in the current view
state. The button label shows the count: "Release All Reviewed WOs (N)".

If a filter is active, the global Release button releases only WOs that match
the filter. The button label and count reflect the filtered scope.

**Hard rule:** Release button scope reflects the current view filter. An
unfiltered view's global Release releases all decided-but-unreleased WOs
across all Projects. A Project-specific Release in a Project header releases
only that Project's decided-but-unreleased WOs.

**Hard rule:** Release does not affect WOs that are still candidates (no
decision recorded). Held-back WOs stay in Stock Fulfillment until the
planner records a decision on them.

### Candidate List

Below the header section, the candidate list shows WOs in default order:
**by Project, then by BOM order within the Project's tree — ancestor
Assemblies displayed before their descendants.**

BOM order in this view is top-down (parent before children). This is
deliberate: when both an Assembly and its descendants qualify as candidates,
the Assembly appears first. The planner evaluates the higher-level pull
before encountering the granular component pulls. Pulling the Assembly
cascades and removes the descendants from the list. Pulling a descendant
first blocks the Assembly from being pulled later (see Defensive Rule below).

Beyond the defensive ordering function, top-down BOM order matches the
physical pulling workflow — a planner pulling for an Assembly considers
the assembly as a whole before its parts.

### Inline Context Expansion

When the planner clicks a candidate row, the row expands inline to show
all other Unreleased WOs that share the same PartID, regardless of which
Project they belong to.

The expansion shows:
- Each competing WO's Project Number and Top-Level Reference
- Demand quantity for each
- Cumulative Demand across all competing WOs
- A "Fulfill from this row" action on each entry (so the planner can fulfill
  another Project's WO from this expansion if they decide a different
  Project should get priority)

The expansion is a context-revealing tool, not a re-ordering tool. The
planner's primary list does not get restructured — they collapse the
expansion and continue down their list. The expansion is an interrogation
moment: "before I fulfill this one, who else wants it?"

Inline expansion works regardless of whether the candidate list is filtered
to one Project or showing all Projects. Even when the list is narrowed to
a single Project, clicking a row reveals competition from Projects outside
the current filter. Filtering scopes the *list*; expansion always reveals
the full cross-Project competition for that PartID.

**Hard rule:** Inline expansion shows the cross-Project competition for a
PartID without moving rows out of their default Project + BOM order
position. This preserves the pulling sequence while supporting cross-
Project judgment calls.

**Hard rule:** Inline expansion is not constrained by active filters. The
expansion always shows all competing Unreleased WOs across all Projects.

## Actions on a Candidate Row

### Fulfill from Stock

The primary action. Only available when Stock Count ≥ WO Demand at the moment
of the action.

**Behavior:**
1. Atomic database transaction:
   - The WO transitions: `Unreleased → Complete`
   - All WorkOrderSteps on the WO transition: `Waiting → Skipped`, with
     a system-generated note "Fulfilled from stock" on each step
   - Stock Count on the Part decrements by WO Demand quantity
   - `stockFulfillmentReviewedAt` is set to current timestamp
   - If the WO's Part is an Assembly with descendant WOs in the same
     Project: cascade applies (see below)
   - AuditLog entries written for: WO state change, every step state
     change, stock count change, every cascade event
2. The row disappears from the candidate list
3. Other candidate rows for the same Part re-evaluate their eligibility:
   any whose demand now exceeds the new stock count are auto-passed-through
   (their `stockFulfillmentReviewedAt` is set, they disappear from the view)

### Pass Through (Explicit)

Available on any candidate row. Used when the planner has reason not to
pull from stock despite eligibility (damaged inventory, miscount suspicion,
allocation concerns, etc.).

**Behavior:**
1. Atomic database transaction:
   - The WO retains `Unreleased` state
   - `stockFulfillmentReviewedAt` is set to current timestamp
   - AuditLog entry written
2. The row disappears from the candidate list
3. The WO will appear in the Batching Lens once its Project clears the gate

### Reconcile Stock

Available on every row in the view. Opens the standard inventory adjustment
modal — the same modal used in any other view that supports stock adjustment
(Parts Master, Distribution Lens). Integration here means the planner does
not need to leave the workflow to fix a discrepancy.

**Modal contents:**
- Part Number + Part Name
- Current Stock Count (read-only display)
- New Stock Count (numeric input, required)
- Reason / Note (text, required)
- Confirm / Cancel

**Behavior on confirm:**
1. Stock Count updated to the new value
2. AuditLog entry written: old value, new value, reason, user, timestamp,
   action type "Stock Reconciliation"
3. The candidate list re-evaluates eligibility for all rows referencing
   this Part — newly-eligible rows may appear, newly-ineligible rows
   are auto-passed-through

**Hard rule:** The displayed Stock Count column on candidate rows is
read-only. Stock Count changes require the explicit Reconcile Stock modal.
This prevents inadvertent edits during fulfillment workflow.

## Assembly Fulfillment Cascade

When a Fulfill from Stock action targets a WO whose Part is an Assembly,
the action cascades to all descendant WOs in the same Project.

**Cascade behavior** (within the same atomic transaction):

1. The fulfilled Assembly WO: `Unreleased → Complete`, all its steps Skipped,
   its Part's Stock Count decrements by the Assembly's Demand quantity,
   `stockFulfillmentReviewedAt` set
2. Every descendant WO in the Project (children, grandchildren, recursively):
   - Transitions: `Unreleased → Complete`
   - All steps transition: `Waiting → Skipped`, with note "Parent assembly
     fulfilled from stock — [Parent Part Number]"
   - `stockFulfillmentReviewedAt` set
   - Stock Count on descendant Parts is **NOT** decremented — those
     components are already inside the fulfilled assembly; they were
     consumed when the assembly was originally built
3. AuditLog entries written for every state change, every step change

**Hard rule:** Fulfilling an Assembly cascades Skip-Complete to all
descendant WOs in that Project's tree. Only the fulfilled Assembly's Part
has its Stock Count decremented. Descendant Part stock is untouched.

**Hard rule:** The cascade applies recursively. If a descendant is itself
an Assembly with its own descendants, those are also Skip-Completed.

**Edge case — descendant was a candidate:** If a descendant WO was itself
appearing as a candidate in this view (because its own stock was sufficient
for its own demand), the cascade pre-empts it. The descendant becomes
Complete via the parent's cascade and disappears from the candidate list.
No separate fulfillment occurs for the descendant — its stock is not
decremented.

## Release Workflow

Release is the explicit action that transitions Pass-Through WOs from
`Unreleased` to `Open`, making them visible to Batching, execution lenses,
Operations Lens, and Project View. It is the planner's affirmative "I'm
done with these" signal for WOs heading to procurement.

Fulfilled WOs do not require release — they transition `Unreleased → Complete`
at fulfillment time and are immediately visible to non-Stock-Fulfillment
views.

### What Releases

When the Release button is clicked, the system identifies all WOs in scope
(global Release: all visible Projects; Project-specific Release: that
Project only) that are eligible to release. A WO is eligible when:

1. It has a decision recorded (`stockFulfillmentReviewedAt IS NOT NULL`)
   AND `WOStatus = Unreleased`, OR
2. It is a non-candidate (Stock < Demand) in `Unreleased` state with
   `stockFulfillmentReviewedAt = null` and a parent Project that's Active

Non-candidates within the scope have their `stockFulfillmentReviewedAt`
stamped at release time as part of the same atomic transaction. This is
the "implicit decision" — the planner clicking Release affirms that the
non-candidates are indeed going to procurement.

### What Does Not Release

- WOs still in candidate state with no decision recorded — these remain
  in Stock Fulfillment until next session (the planner held them back
  intentionally)
- Fulfilled WOs (already Complete) — they were never Unreleased after
  fulfillment, so there's nothing for Release to do
- WOs in Projects whose Status is no longer Active (e.g., the Project was
  archived mid-session — defensive case)

### Release Transaction

A single atomic transaction:
1. Identify all WOs in scope that are eligible to release (per above)
2. For non-candidates that don't yet have a decision marker: stamp
   `stockFulfillmentReviewedAt` with current timestamp
3. For all eligible WOs: transition `WOStatus: Unreleased → Open`
4. Write a single aggregate AuditLog entry with action type
   `StockFulfillmentReleased`: count of WOs released, scope (global vs.
   Project-specific with Project ID), user, timestamp
5. Per-WO state changes are derivable from the aggregate event timestamp
   — no separate per-WO AuditLog entries are written for the release
   action itself

### Visibility After Release

A released WO (now `WOStatus = Open`) becomes visible to:
- **Batching Lens** — for procurement and execution scheduling
- **Operations Lens** — for shop-floor visibility
- **Project View** — for project-level visibility
- **Execution lenses** (Purchasing, Receiving, Machining, etc.) — when
  step prerequisites are met

A released WO disappears from Stock Fulfillment because the entry condition
`WOStatus = Unreleased` no longer holds.

**Hard rule:** Release is one-way. A WO transitioned from Unreleased to
Open does not return to Unreleased. WOs cannot return to Stock Fulfillment.

**Hard rule:** Released WOs remain associated with their Project for all
operational and historical purposes. Release only changes WOStatus, not
the WO's relationships.

## Auto-Pass-Through Behavior

Auto-pass-through is the system's mechanism for stamping
`stockFulfillmentReviewedAt` on WOs that lose candidacy without explicit
planner action.

**Trigger conditions:**
1. **Stock dropped below demand mid-session.** A candidate WO loses
   eligibility because another candidate's fulfillment consumed available
   stock. The WO is auto-passed-through immediately
2. **Reconciliation reduced stock below demand.** A Reconcile Stock action
   that lowers the count to a level that a candidate row can no longer
   satisfy — that candidate is auto-passed-through
3. **Release event with non-candidates in scope.** When the planner clicks
   Release, any non-candidates within scope that don't yet have a decision
   marker get auto-passed-through as part of the release transaction

**Behavior (in cases 1 and 2):**
- WOStatus remains `Unreleased`
- `stockFulfillmentReviewedAt` set to current timestamp
- AuditLog entry written with action type `StockAutoPassThrough` and reason
  ("stock depleted by other fulfillment", "reconciliation reduced stock")
- Row disappears from the candidate list, but the WO remains pending Release

**Behavior (in case 3):**
- `stockFulfillmentReviewedAt` set as part of the release transaction
- WOStatus transitions `Unreleased → Open` as part of the release transaction
- AuditLog: covered by the aggregate `StockFulfillmentReleased` event

**Hard rule:** Auto-pass-through during a session (cases 1 and 2) only
records the decision. Transition to Open requires the explicit Release
button click.

**Hard rule:** Auto-pass-through is logged distinctly from explicit
Pass Through in AuditLog. Both result in the same WOStatus, but the
audit trail preserves which was system-driven vs. planner-driven.

## Candidate List Columns

| Column | Notes |
|--------|-------|
| Project | Project Number + Top-Level Reference (e.g., "20137.01"). Includes the Project's color indicator |
| Part Number | Monospace, links to Part Form |
| Part Name | |
| BOM Position | Path within the Project's BOM tree (e.g., "20137.01 → BRACKET-100 → FAST-M5"). Informs the planner where to physically allocate the pulled parts — pulled components go into bins associated with their parent Assembly |
| WO Demand | Quantity required by this WO |
| Stock Count | Read-only display. Live-updates as actions are taken |
| Cumulative Demand | Total Unreleased demand for this PartID across all Projects. Surfaces cross-Project context at a glance |
| Due Date | Helps disambiguate competing WOs when stock is constrained |
| Action — Fulfill | Button. Available when Stock ≥ Demand at moment of action |
| Action — Pass Through | Button. Always available |
| Action — Reconcile Stock | Modal trigger. Always available |

**Visual conventions:**
- **Row shading distinguishes Part from Assembly** (consistent with Parts
  Master spec). This communicates Part type without consuming horizontal
  column space — important because the Assembly cascade behavior makes
  Part type significant
- The Project color indicator on the Project column reinforces visual
  Project grouping when the unified list is in use
- The Stock Count value updates live as actions are taken in the session.
  When stock drops below demand for a row, the row exits the candidate
  list (auto-pass-through) rather than displaying as ineligible

**Columns deliberately excluded:**
- *Priority* — Stock Fulfillment is about availability, not urgency.
  Priority is set downstream and is more WO-specific
- *Routing* — Irrelevant to fulfillment decisions
- *State / Sub-Status* — All candidate rows are Unreleased with no step
  activity by definition
- *Reviewed Indicator* — Candidate rows are by definition unreviewed; the
  marker is meaningful only in retrospective views

## Recovery Paths

**Missed fulfillment** — a WO should have been pulled from stock but was
passed through, and is now stuck in execution waiting for procurement that
shouldn't be needed:
- A manager uses **Project View** to skip-and-fulfill the affected WO. This
  is a deliberate manager action that completes the WO using stock that was
  always available. Stock decrements at the time of the recovery action,
  AuditLog records the recovery
- This recovery path lives in Project View, not in this view

**Wrong fulfillment** — a WO was fulfilled from stock that turned out to be
incorrect (insufficient physical stock, wrong part, etc.):
- **Not in Rev 1 scope.** No reversal workflow. The recorded fulfillment
  stands; if the physical situation differs, the planner reconciles stock
  manually (which produces an audit trail) and the resulting shortage is
  managed downstream
- This is a known Rev 1 limitation. Reversal workflows are a Rev 2 backlog item

## Schema Implications

This view requires the following schema additions, applied during Stage 6:

| Field | Type | Notes |
|-------|------|-------|
| `WorkOrder.stockFulfillmentReviewedAt` | DateTime, nullable | Stamped when WO has a decision recorded (Fulfill, explicit Pass Through, auto-Pass-Through, cascade-skip, or implicit decision at release time for non-candidates) |
| AuditLog action types | Enum or string convention | Add: `StockFulfilled`, `StockPassThrough` (explicit), `StockAutoPassThrough`, `StockReconciliation`, `AssemblyCascadeSkip`, `StockFulfillmentReleased` (aggregate release event) |

The `Skipped` value on `StepState` (already pending Stage 6 from OQ-013)
is required for this view to function. Steps fulfilled-from-stock or
cascade-skipped use this state.

**Visibility rule for execution lenses and operational views (Project View,
Operations Lens, Batching Lens, Purchasing, Receiving, etc.):** these views
filter out WOs with `WOStatus = Unreleased`. WOs in `Unreleased` state are
visible only in Stock Fulfillment, Project Creation View (for Project
Summary context), and Project View (under appropriate planning-state
visual treatment).

The existing `WOStatus` enum (`Unreleased | Open | Complete`) carries the
visibility distinction directly — no separate `releasedAt` field is needed.
Fulfill from Stock transitions `Unreleased → Complete`; Release transitions
`Unreleased → Open`.

## Design Notes

- This view exists because of a specific operational pattern: a small shop
  pulls from stock as a discrete planning act, not as a continuous inventory
  query. The act of "going through what we already have" before "ordering
  what we need" is a real workflow step. The view honors that step
- The view is deliberately narrow. Surfacing only fulfillment candidates
  reduces cognitive load for the planner and aligns with Tirion's
  principle of equipping people to make decisions rather than confirming
  decisions the system can already determine
- The Reconcile Stock integration is the "free audit" pattern — the planner
  is already touching inventory, so making reconciliation available without
  a context switch is high value at low cost
- The inline expansion pattern for cross-Project competition keeps the
  pulling sequence intact while supporting deliberate cross-Project judgment
  calls. This is a UX pattern that may apply to other Tirion views where
  context matters but row order is operationally meaningful
- Auto-pass-through being logged distinctly from explicit Pass Through is
  important for retrospective analysis. "Why did this WO end up in
  procurement?" should be answerable from the audit trail without ambiguity
- The Assembly cascade is structurally significant. It means Stock Fulfillment
  is the only place in the system where one WO action can cause many WO state
  transitions in a single transaction. Implementations need to handle this
  correctly under all conditions (deep trees, mixed Part/Assembly descendants,
  in-flight competing fulfillments)
- The BOM Position column is more than orientation — it informs physical
  allocation. The shop floor workflow assumes pulled components are
  physically organized into bins associated with their parent Assembly.
  A row showing "20137.01 → BRACKET-100 → FAST-M5" tells the planner the
  pulled fasteners go in the BRACKET-100 bin. Removing this column to save
  space would force planners to re-derive this context from elsewhere
- The top-down BOM ordering is a defensive design feature, not just a
  display preference. Presenting an Assembly before its descendants
  encourages the planner to evaluate the higher-level pull first. Combined
  with the descendant-then-ancestor lock, this minimizes the chance of
  duplicated pulls in normal usage
