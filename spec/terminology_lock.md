# Tirion — Terminology Lock

This document is the authoritative definition of all domain terms used in Tirion.
Every spec file, every code comment, every developer conversation must use these
terms consistently and precisely.

**This document is locked.** Terms are not redefined elsewhere. If a downstream
spec reveals a genuine conflict with a definition here, that conflict is resolved
by updating this document first, then propagating the change.

---

## Cluster 0 — Cross-Cutting Terms

### Production Row
The active unit of work in a process lens. A Production Row is a Production Batch
when the underlying Work Orders are batched, or a Work Order when unbatched. All
lens actions, state transitions, and blocker operations act on the Production Row.
Individual WO details remain accessible within a batched Production Row but the
Batch is what the operator sees and acts on.

**What it is NOT:** Production Row is not a database entity. It is the term for
whichever object — WO or Batch — is currently the active execution unit in a lens.

---

## Cluster 1 — Core Entities

### Work Order
A granular demand row representing the requirement to produce or procure one Part
or Assembly for one specific Project. Created automatically when a Project is
generated by walking the BOM tree. Never refers to grouped or consolidated work.

**What it is NOT:** A Work Order is never called a batch. When both a Work Order
and a Production Batch could be the subject of a sentence, name which one explicitly.

**Hard rule:** A Work Order is the primary execution unit when unbatched. When a
WO is assigned to a Production Batch, the Batch becomes the primary execution unit
and is what process lenses display.

---

### Production Batch
A grouping of two or more Work Orders for the same PartID, consolidated for
execution efficiency. The primary execution unit in all process lenses when WOs
are batched. Never refers to an individual demand row.

**What it is NOT:** A Production Batch is never called a work order. It is not a
single-WO concept — a batch with only one member dissolves back to a standalone WO.

**Hard rule:** All member WOs must share the same PartID and identical routing
template. A batch dissolves if membership drops to one WO.

**Hard rule:** Batch priority is derived as the highest priority among member WOs.
Batch due date is derived as the earliest due date among member WOs. Both
re-derive whenever membership changes.

---

### Part
A leaf node in the BOM. Always procured or manufactured. Routing always begins
with Purchase and Receive steps. Has no children in the BOM. Shares a database
table with Assembly, distinguished by `partType = Part`.

**What it is NOT:** A Part does not have children in the BOM. If a component has
children, it is an Assembly.

---

### Assembly
A node in the BOM that has one or more children. Never uses Purchase or Receive
steps. Readiness for downstream routing steps depends on child Work Order
completion. May itself be a child of another Assembly. Shares a database table
with Part, distinguished by `partType = Assembly`.

**What it is NOT:** An Assembly does not go through Purchase or Receive steps.
Assembly blockers apply only to the Assembly's own routing steps — not to child WOs.

**Hard rule:** Assembly routing templates may never include Purchase or Receive
steps. Enforce at template assignment time on the Part record, not at template
creation time.

**Hard rule:** An Assembly WO's first routing step is Ready only when all child
WOs are Complete. Subsequent Assembly steps follow the standard prerequisite
rule — all steps with a lower index on the same WO must be Complete.

---

### Project
A top-level production initiative that drives Work Order generation. In Rev 1,
always represents a customer order. In Rev 2, extended to cover stock fulfillment
runs via a `projectType` field. The WO generation logic, BOM walking, and
execution tracking are identical regardless of project type.

**What it is NOT:** A Project is not a single Work Order. It is the container
that holds an entire WO tree generated from a BOM.

**Hard rule:** `customerName` remains nullable at all times. No constraint may
assume every Project has a customer. This protects the Rev 2 extension.

**Rev 2 note:** Add `projectType` enum (`CustomerOrder | StockFulfillment`).
No other schema changes required.

---

## Cluster 2 — Routing and Process Concepts

### Routing Template
A named, reusable sequence of process steps defined at the Part level. The
persistent source of truth for how a category of parts is produced. When a
template is edited, the change applies immediately to all Parts that reference
it. All future Work Orders generated from those Parts use the updated sequence.
Open Work Orders already in progress are not modified — they are flagged for
manual review. A human resolves each flagged instance deliberately.

**What it is NOT:** A Routing Template is not per-project or per-WO. It is a
definition-layer record. Changes to it do not cascade to open Work Orders.
This is Principle 10 and is non-negotiable.

**Hard rule:** Editing a template requires a confirmation screen showing the
count of Parts that reference it and a warning that open Work Orders will be
flagged. No silent edits.

**Hard rule:** Template changes never cascade to open Work Orders (Principle 10).

---

### Routing Template Step
One step within a Routing Template, with a sequence position (1-based) and a
Process Type. The ordered sequence defines the production path for all Parts
referencing this template. No enforced maximum step count — practical limit is
the number of visually distinct Process Type colors available in the UI.

---

### Work Order Step
A snapshot copy of a Routing Template Step that lives on a specific Work Order.
Created at WO generation time. Permanently independent of the template after
that point. State is tracked here — not on the template. This is the actual
unit of execution on the shop floor.

**Hard rule:** Work Order Steps are generated as a snapshot at WO creation.
Template changes after that point do not affect existing Work Order Steps.

---

### Process Type
The category of work performed at a routing step. Each Process Type has an
assigned color used as a visual identifier across all lenses. ProcessTypes
are locked in Rev 1 (view-only via the configuration surface — see
`configuration_management_spec.md`).

**Rev 1 Process Types (9 active):**
| processCode | processName | Has Execution Lens | Anchor Eligible |
|-------------|-------------|--------------------|-----------------|
| PURCHASE | Purchase | Purchasing Lens | Yes |
| RECEIVE | Receive | Receiving Lens | Yes |
| MACHINE | Machine | Machining Lens | Yes |
| WELD | Weld | No (managed via Operations Lens) | Yes |
| BLACKEN | Blacken | No (managed via Operations Lens) | Yes |
| PAINT | Paint | No (managed via Operations Lens) | Yes |
| PRINT_3D | 3D Print | No (Rev 2 lens planned) | Yes |
| ASSEMBLE | Assemble | Assembly Lens | Yes |
| DISTRIBUTION | Distribution | Distribution Lens | Yes |

The canonical Rev 1 seed list lives in `seed_data_spec.md`.

**Lens-having vs. anchor-eligible:**
All 9 Rev 1 ProcessTypes are anchor-eligible in management views (Operations
Lens, Project View). Five have dedicated execution lenses; the other four
(Weld, Blacken, Paint, 3D Print) are managed via Operations Lens / Project
View. Anchor + filter support serves as the substitute working surface for
non-lens processes — see `anchor_filter_spec.md`.

**Operational notes for non-lens processes:**
- **Weld and Paint:** Performed by computer-illiterate operators; management
  mediates state updates verbally
- **Blacken:** Outside-vendor process; management updates state based on
  shipping/receiving knowledge
- **3D Print:** Rev 2 dedicated lens planned

**Note:** Inspect and Finish are NOT Rev 1 ProcessTypes. They lack execution
lenses and would create orphaned process types that routing templates could
reference but no surface could operationalize. They may be added in Rev 2
alongside lens definitions.

**Note:** StockFulfillment is NOT a ProcessType. It is a planning gate, not
a routing process.

**Hard rule:** Adding new ProcessTypes is a Rev 2+ feature paired with the
development work to make new types operationally meaningful (lens
definitions or management workflow definitions, side panel content, etc.).

---

### External Process

This concept has been **simplified in Rev 1.** Earlier drafts had an
`isExternal` flag on ProcessTypes to identify processes that physically
leave the shop. The flag has been removed; the system handles external
processes (Blacken in Rev 1) via the standard step state machine plus
sub-status vocabulary. Step states (Ready, Started, Complete) plus
intermediate sub-statuses (e.g., "At Vendor", "Received") capture the
external lifecycle without requiring a separate field.

External processes may evolve in Rev 2 alongside outside-vendor receiving
modes and the related material handling work. If a structural distinction
becomes necessary then, the `isExternal` field can be re-introduced.

---

### Distribution
The final routing step. The physical and digital act of resolving completed parts
to their correct destinations — moving parts into project bays and stock
locations, and updating the system's count of finished work accordingly.

**Hard rule:** Distribution is always the final step in any routing sequence.
No steps may follow it.

---

## Cluster 3 — Demand and Quantity Concepts

### Demand Quantity
The quantity required for a given row. At the Work Order level, the quantity of
this Part required for this Project. At the Production Batch level, the sum of
member WO demand quantities. Demand is the hard floor — CompletedQty must meet
or exceed it for work to proceed downstream without manager intervention.

**Hard rule:** Scrap never satisfies demand. Only good CompletedQty counts.

**Hard rule:** If CompletedQty falls below Demand at any step where quantity is
required, a blocker is created. A manager resolves it via one of the available
resolution paths. See Blocker Resolution.

---

### Planned Quantity
The planner's production target for a Work Order or Batch. Always equal to or
greater than Demand Quantity. Represents intentional overage — extra parts for
stock, buffer, or setup efficiency. In Rev 1, set manually by the planner.

**What it is NOT:** Planned Quantity is not a ceiling. CompletedQty does not
need to match it — it only needs to meet or exceed Demand Quantity.

**Rev 2 note:** May be informed by minimum and maximum bin sizes in a later
revision.

---

### Completed Quantity
Good parts produced, recorded at each relevant step completion. Must meet or
exceed Demand Quantity to proceed without intervention. No upper bound —
CompletedQty may exceed Planned Quantity at operator discretion. Applies at
any step where physical output is produced — not restricted to a specific
Process Type.

**Purchase and Receive steps:** CompletedQty is nullable at Purchase and Receive
steps in Rev 1. These steps often involve raw material received as continuous
stock (bar, sheet, tube) that is cut into discrete units later. Recording
CompletedQty is available but not required at these steps. Required at all
downstream process steps.

---

### Scrap Quantity
Defective or unusable parts produced during a step. Always recorded alongside
CompletedQty at step completion. Never counts toward satisfying Demand. Logged
for visibility and future analysis.

---

### Cumulative Demand
The total Demand Quantity for a Part across all open Work Orders, regardless of
Project. Used in the Purchasing Lens as a reference for total buy quantity.

**Rev 1 scope:** Simple aggregate across open WOs. Precise net unfulfilled demand
calculation (excluding on-hand, allocated, and on-order quantities) is a Rev 2
refinement dependent on the full material handling model.

---

### Stock Count
The quantity of a Part currently available on hand in stock. Stock is always
physically present and unallocated. When a part is allocated to a project, it
is physically moved to project staging and is no longer counted as stock.
Updated at Distribution when completed parts are resolved to their destinations.

**What it is NOT:** Stock Count does not include parts allocated to a project
or parts in transit. Allocated parts have already left stock.

---

## Cluster 4 — State and Status Concepts

### State
The execution condition of a Work Order Step, Work Order, Production Batch, or
Blocker. Stored in the database. Drives what lenses display and what actions
are available.

---

### Sub-Status
Free text operator context within a state. Does not change state. Does not
trigger system behavior. Gives operators a way to communicate real-world
condition without formal state transitions.

**Rev 2 note:** May offer a per-process validated dropdown with common values
while retaining free text as an option.

---

### Waiting
A step that cannot yet be worked because prior steps are not complete. The
default state for all steps except the first when a WO is released into queues.

**What it is NOT:** Waiting does not apply to unreleased Work Orders. An
unreleased WO is invisible to process queues entirely — it does not appear
as Waiting in any lens.

---

### Ready
A step where all prerequisites are met and work can begin. Always derived by
application logic — never set directly.

**Derivation rules:**
- Standard step: all steps with a lower stepIndex on the same WO are Complete
- Assembly first step: all child WOs are Complete
- Assembly subsequent steps: all steps with a lower stepIndex on the same WO
  are Complete (standard rule applies after the first step)

**Hard rule:** Ready state is never set directly. It is always computed.

---

### Started
Work is actively in progress on this step. An optional state — work may
transition directly from Ready to Complete without passing through Started.

---

### Complete
The step has been finished. CompletedQty and ScrapQty have been recorded where
required. For the final step, WO status updates to Complete in the same
database transaction.

**Hard rule:** WO completion is transactional. Final step completion and WO
status update happen together. No partial updates.

---

### Blocked
Work cannot proceed due to an exception. Requires explicit creation with a note.
Requires explicit resolution with a note. Never resolves silently.

**Hard rule:** Blocking a batched WO blocks the Batch, not the individual WO.
Assembly blockers apply only to the Assembly's own routing steps, not to
child WOs.

---

### Released (Stock Fulfillment Release)
A Work Order has been processed through Stock Fulfillment with a decision
recorded. Either the WO was Fulfilled from Stock (transitioned directly to
Complete via Skipped steps), or it was Pass Through (decision marker
`stockFulfillmentReviewedAt` set, WOStatus remains Unreleased, becomes visible
to Batching).

Release is the first of two gates between compilation and execution. It does
not transition WOStatus by itself for Pass-Through WOs — it sets the marker
that makes the WO visible to the second gate (Batching).

**Hard rule:** Stock Fulfillment Release is per-WO, not per-Project (Model 1).

---

### Confirmed (Batching Confirm)
A Work Order has been processed through Batching Lens with a Confirm action,
transitioning WOStatus from Unreleased to Open. This is the second of two
gates between compilation and execution. Confirmation makes the WO visible
to execution lenses.

A WO can be Confirmed as a member of a Production Batch (via batch confirm)
or as a standalone Production Row (via singleton confirm).

**Hard rule:** Confirmation is one-way — once a WO is Open, it does not
return to Unreleased. Restructuring of Open work uses the Batch Adjustment
Workspace via the Blocker workflow.

---

### Skipped
A WO Step state indicating the step was bypassed without being executed.
Used in the Stock Fulfillment workflow:
- When a WO is Fulfilled from Stock, all its routing steps transition to
  Skipped in the same atomic transaction
- When an Assembly is Fulfilled from Stock, the cascade-skip behavior applies
  Skipped state to all descendant WO routing steps as well

Skipped is functionally equivalent to Complete for downstream calculations
(an upstream Skipped step satisfies the Ready prerequisite for downstream
steps), but distinguishes the audit trail — a Skipped step did not produce
parts via the standard process; the parts came from existing stock.

**Hard rule:** WO Step Ready derivation includes Skipped as a satisfying
upstream state (alongside Complete).

---

### Cancelled
A Work Order state indicating the WO has been administratively retired.
Cancelled is a top-level WOStatus value — it replaces the prior status, not
an additional flag layered on top. There is no "Open + Cancelled" or
"Unreleased + Cancelled."

Set via the Cancel primitive (Manager/Admin action on leaf WOs only). One-way
in Rev 1 — there is no un-Cancel workflow.

Cancelled WOs are excluded from operational visibility, readiness math,
demand calculations, and Project Status auto-transition logic. They remain
in the database for historical/audit purposes.

**Hard rule:** Cancel is available only on leaf WOs (no non-Cancelled
descendants). The user must walk the BOM tree leaf-first to cancel an
Assembly with descendants.

---

## Cluster 5 — BOM and Component Concepts

### BOM (Bill of Materials)
The complete definition of what components make up an Assembly.
Project-agnostic — the BOM defines relationships at the Part definition level,
not per project. When a Project is created, the system walks the BOM tree and
generates a mirrored Work Order tree.

---

### BOM Row
One record representing a single parent/child relationship. One row per unique
child within a parent Assembly. A child may not appear twice under the same
parent — quantity is consolidated on one row. A child may appear under multiple
different parent Assemblies.

**Hard rule:** Duplicate children under the same parent are blocked at the
database and form level.

---

### Parent Assembly
An Assembly that contains one or more children in the BOM. Always an Assembly —
Parts cannot have children in the BOM.

---

### Child Component
A Part or Assembly that appears as a child in a BOM row. May appear under
multiple different parent Assemblies. When the child is itself an Assembly,
it has its own children, creating a recursive tree.

---

### BOM Tree
The full recursive structure of an Assembly and all its descendants. Depth is
unlimited — no enforced maximum. A slow system is preferable to one that
cannot represent a valid BOM. Circular references are explicitly prohibited.

**Hard rule:** Circular references are never allowed at any depth. An Assembly
cannot be its own ancestor. Enforce via full ancestry traversal on every
Add Component operation. This is a correctness requirement, not optional.

---

### Component Readiness
A derived ratio representing how many child Work Orders under an Assembly WO
are Complete versus total children. Expressed as a count (e.g., 11/15). Used
in the Assembly Lens as the primary indicator of build readiness. Never stored —
always computed.

---

### Effective Quantity
The quantity of a child component required, multiplied up through all ancestor
levels. Calculated as parent quantity multiplied by child quantity at each level
of the BOM tree. Used when generating Work Order quantities from the BOM tree
during Project creation.

---

## Cluster 6 — Purchasing and Receiving Concepts

### Supply Order
A lightweight procurement reference record representing a specific expected
delivery from a vendor. Corresponds to one expected receipt event — one
shipment, one delivery date, one tracking reference. Not a formal purchase
order document — covers any procurement action including online retailers,
distributors, and formal vendors equally.

**What it is NOT:** A Supply Order is not a primary workflow object in Rev 1.
No workflow logic depends on Supply Orders in Rev 1. The Production Row is
the primary object in both Purchasing and Receiving lenses.

**Rev 1 behavior:** Created manually by buyer as a grouping reference for
orders sharing delivery characteristics. ETA, tracking number, vendor, and
order reference are fields on the Production Row's purchasing step as well
as on the Supply Order.

**Rev 2 note:** Supply Orders become full workflow objects with partial receipt
management, multi-line tracking, and vendor performance reporting. The Rev 1
schema supports this without structural rework.

**Hard rule:** ETA is always editable regardless of state. Every change logged
to AuditLog with user and timestamp.

**Hard rule:** One Supply Order = one expected receipt event. If one vendor
order has items arriving on different dates, create one Supply Order per
delivery group.

---

### Supply Order Line Allocation
The junction record linking a Supply Order Line to one or more Work Orders.
Replaces the direct one-to-one `workOrderId` FK on the original PurchaseOrderLine.
One Supply Order Line for raw material may satisfy multiple Work Orders.

---

### RFQ (Request for Quote)
An optional pre-commitment activity in the purchasing workflow. A request sent
to a vendor for pricing before committing to a Supply Order. RFQ state lives
as a sub-status on the WorkOrderStep, not on a Supply Order — because the
outcome of an RFQ may result in multiple Supply Orders, a different vendor,
or different quantities than originally anticipated.

**What it is NOT:** RFQ is not tied to a Supply Order. It precedes the decision
to create one.

---

### Purchasing Sub-Status
Operator context on the Purchase WorkOrderStep. Validated dropdown per the
ProcessType sub-status vocabulary. Never drives state transitions.

| Sub-Status | Meaning |
|------------|---------|
| Stock Checked — No Material | Physically verified not on hand. Needs to be ordered |
| RFQ Pending | Quote requested from vendor. Awaiting response |
| Quote Received | Vendor responded. Buyer evaluating |
| Order Placed | Submitted to vendor. ETA recorded on Production Row |
| Delayed | ETA passed or revised. Note field captures context |

Blank sub-status means stock check has not yet been performed.

**Fulfill from Stock** is a state transition — not a sub-status. It completes
the Purchase step entirely and removes the Production Row from the purchasing
queue when on-hand stock satisfies demand.

---

### Default Vendor
The preferred supplier assigned to a Part record. Used as the purchasing default
but overridable at the WO level for a specific order. Subject to Principle 10 —
if the Default Vendor changes on a Part record, WOs already in Order Placed
state or later are flagged for manual review rather than automatically updated.

---

### Material Specification (MaterialSpec)
A definition of a raw material type including material name, form, stock size,
and unit of measure. Referenced by Parts. The primary grouping dimension in the
Purchasing Lens — demand is grouped by MaterialSpec and Vendor to surface total
material requirements in one place and support consolidated purchasing decisions.

---

### Receiving
The completion event for the Receive WorkOrderStep. Not a separate workflow
object in Rev 1 — it follows the standard WorkOrderStep state model. When the
receiver marks the Receive step Complete, they may confirm or update the
CompletedQty entered by purchasing, enter it if left null, or leave it null.
CompletedQty is never required at the Receive step in Rev 1.

**Receiving Sub-Status:**
| Sub-Status | Meaning |
|------------|---------|
| Expected | ETA known. Shipment on its way |
| Delayed | ETA passed or updated. Note field captures context |
| Partial | Some quantity received. Remainder pending (note only in Rev 1) |

**Rev 2 note:** Full Receipt workflow with partial receipt management, refused
quantity tracking, putaway confirmation, and vendor performance activates in
Rev 2. Receipt table exists in Rev 1 schema as a reference record.

---

### Putaway
The physical act of moving received goods to their correct destination after
receipt is confirmed. Destination is determined by routing context:
- Parts with further processing steps → next process station for staging
- Finished purchased components with no further steps → project staging
  (Distribution step is effectively completed at Putaway)
- Excess → stock

**Hard rule:** Finished purchased components that require no further processing
have their Distribution step completed at Putaway. They do not appear in the
Distribution Lens.

---

### Refused Quantity
Items rejected at receiving — wrong items, damaged goods, or incorrect
specification. Never enters stock or satisfies demand. Creates an exception
record for purchasing follow-up. Rev 2 feature — noted here for schema awareness.

---

## Cluster 7 — Blocker and Exception Concepts

### Blocker
An explicit record that work cannot proceed due to an exception. Always requires
a creation note. Always requires a resolution note. Never resolves silently.
Applies at the Batch level when a WO is batched, at the WO level when unbatched.

**Hard rule:** Assembly blockers apply only to the Assembly's own routing steps.
Child WOs are not affected by a parent Assembly blocker.

**Hard rule:** No blocker may be created without a note. No blocker may be
resolved without a note.

---

### Blocker Category
A classification of the blocker's scope. Defaults to Local.

- **Local** — The issue can be resolved within the originating process or station.
- **Escalated** — The issue requires management involvement beyond the originating
  process.

---

### Blocker Status
The current condition of a Blocker record.

- **Open** — Created, not yet acknowledged or acted on. Displays as an active
  exception in all lenses.
- **Pending Resolution** — Deliberately acknowledged by a manager. The cause is
  known and the team is waiting for an external condition to resolve before acting.
  Displays as acknowledged but unresolved. Work cannot proceed.
- **Resolved** — Closed via an explicit resolution type with a required note.

**Hard rule:** Pending Resolution is not resolved. Work cannot proceed. It
communicates intentional waiting, not completion.

---

### Blocker Resolution
The explicit act of closing a blocker. Always requires a note. Resolution types:

**Cleared**
The issue is resolved with no structural change to the WO or Batch. State returns
to its pre-blocker condition. Available to Operators at their own station,
Managers, and Admins.

**WO Split**
The blocked WO is divided into two WOs. The original WO retains the CompletedQty
already produced and proceeds downstream. A new WO is created for the remaining
demand. Parent Assembly WO demand relationships are preserved — the parent sees
the combined demand of both resulting WOs. Manager and Admin only. Rev 1 feature.
Full implementation spec to be developed before Phase 6 build.

**Batch Adjustment**
A blocked Batch is restructured by reassigning member WOs to other batches or
leaving them unbatched until the blocked batch reaches a valid state. Uses the
Batch Adjustment Workspace. Manager and Admin only. Only available when the
blocked entity is a Batch.

**Routing Rollback**
The blocked entity's routing is reset. All steps downstream of the current step
return to Waiting. Used when work must be redone from an earlier point. Manager
and Admin only.

**Demand Adjustment**
Deferred to Rev 2. In Rev 1, almost every WO has a parent Assembly dependency,
making this resolution invalid in nearly all cases.

**Rev 2 rule:** A manager reduces the Demand Quantity on a blocked WO to match
actual production output. Only available when no parent Assembly WO depends on
this WO's demand. Demand may only be adjusted downward as a blocker resolution.
Manager and Admin only.

---

### Batch Adjustment Workspace
A dedicated manager interface for restructuring a blocked Batch. Operates on a
working draft — no changes commit until the manager explicitly confirms the final
state. Exiting without confirming discards all provisional changes. The workspace
shows live demand validation as WOs are reassigned so the manager always sees
whether the current draft produces a valid end state.

---

### WO Split
An exceptional management action that divides one Work Order into two. The
original WO retains the CompletedQty already produced and proceeds downstream.
A new WO is created for the remaining demand. Parent Assembly WO demand
relationships are preserved. Not available as a standard operator action.
Rev 1 feature. Full implementation spec to be developed before Phase 6 build.

---

### Demand Adjustment
Deferred to Rev 2. See Blocker Resolution — Demand Adjustment above.

---

### Assembler Review Flag
A lightweight feedback mechanism available in the Assembly Lens. Creates a flag
record with an optional note. Does not block production. Does not change WO or
step state. Not a Blocker. Used by assemblers to surface real-world issues —
interference fits, unclear drawings, missing callouts — for later management or
engineering review.

**Distinct from Definition Change Flag.** The Assembler Review Flag is created
by the assembler at the time of work; it represents an issue surfaced during
execution. The Definition Change Flag (see below) is created by the system in
response to definition edits. They use different schemas, surfaces, and
resolution workflows.

**Rev 1:** Flag creation only.
**Rev 2:** Management review workflow and resolution tracking.

---

### Definition Change Flag
A persistent system-created record per (definition change × affected entity)
pair, surfacing the impact of definition-layer edits to active Work Orders or
Production Batches. Created automatically when a Part, BOM, or Routing Template
edit affects open work; resolved manually by a Manager or Admin via Dismiss
(accept drift) or Accept Change (apply the change to the affected entity).

The Definition Change Flag system is the mechanism by which Principle 10
(definition changes don't auto-cascade to WIP) surfaces deferred reconciliation
decisions for human resolution.

Visible as a yellow flag indicator on affected WO and Batch rows across all
operational lenses. See `definition_change_flag_spec.md` for the full
specification.

**Distinct from Assembler Review Flag** (above) and from Blocker (which stops
work). Definition Change Flags are informational and decision-pending; they
do not block work.

**Hard rule:** Resolution requires a note. Permissions are Manager/Admin only.
Operators see the flag indicator but cannot resolve.

---

### Project Archival
A manager action that sets a Project status to Archived. All associated WOs
are removed from active process queues. WOs currently assigned to a Production
Batch are removed from that Batch — if the Batch drops to one member, it
dissolves. All archival actions logged to AuditLog. The system does not attempt
to resolve or account for WIP already in progress at the time of archival.

**Hard rule:** Project Archival requires explicit manager confirmation.
Not available to Operators.

---

## Cluster 8 — Primitives (Manager-Level Edit Actions)

The five primitive actions Manager/Admin users can perform to edit WIP state.
Each is a one-way action (or, where reversible, requires explicit re-execution
of an inverse).

### Loss
Reduce CompletedQty on a step by moving units to ScrapQty. Used when good
parts are subsequently identified as defective. Manager/Admin only. Requires note.

### Rollback
Revert a step's state from Complete back to Started or Ready. Used when a
step was incorrectly marked Complete or when downstream issues require
reworking the step. Manager/Admin only. Requires note.

### Return to Stock
Reverse a Fulfill from Stock action — increment stock count, transition WO
from Complete back to Open with steps reset to Waiting. Used when stock-fulfilled
parts are no longer needed for the original WO. The Assembly cascade reverse
(if the fulfilled WO was an Assembly) restores descendants from Skipped state.
Manager/Admin only. Requires note. See Stock Fulfillment spec for details.

### WO Split
Divide one Work Order into two new WOs. The original WO is archived; the two
new WOs inherit its identity and state appropriately based on the split point.
Used to isolate defects, partial completion, or to enable independent handling
of portions of a WO. Manager/Admin only. Requires note. Full specification in
forthcoming `wo_split_spec.md`.

### Cancel
Set a leaf WO's WOStatus to Cancelled, removing it from operational visibility,
readiness math, demand calculations, and Project Status auto-transition logic.
Available only on leaf WOs (no non-Cancelled descendants). One-way in Rev 1.
Manager/Admin only. Requires note. See `definition_change_flag_spec.md` (Cancel
section) for details.

**Hard rule:** Cancel is leaf-first only. The user must walk the BOM tree
leaf-first to cancel an Assembly with descendants. This prevents bulk
cancellation accidents and forces case-by-case decisions about how to handle
in-flight work.

## Cluster 9 — UI and Interaction Concepts

### Composition Column
The leftmost column of the Batching Lens grid (Column A). A specialized column
unique to the Batching Lens that serves as the planner's working canvas for
draft batch composition. Each row's cell in this column acts as a container
for chips representing Unreleased+reviewed candidate WOs being placed into
draft batches.

Open rows have empty Composition Column cells by default — Open batch members
are not represented as chips because they cannot be moved.

See `batching_lens_spec.md` for the full Composition Column behavior including
chip anatomy, drag mechanics, and confirm toggle hierarchy.

### Chip
An interactive UI token in the Batching Lens Composition Column representing
one Unreleased+reviewed candidate WO. Displays Project Color, Project Number +
Top-Level Reference, and Demand Quantity. Persistent during a session and
movable between any eligible Composition Column cells.

### Bright Blue Signaling
A visual convention in the Batching Lens where row attribute values that have
been modified by draft batch composition (Demand Quantity, Priority, Due Date,
etc.) display in bright blue. Provides immediate visibility of what will
change at draft confirmation.

### Row Indicators
Three flag-colored icons that surface row state across operational grids
and side panels. Each indicator has a distinct color, hover behavior, and
click action.

| Color | Indicator | Surfaces When | Hover Reveals | Click Action |
|-------|-----------|---------------|---------------|--------------|
| **Red** | Blocker | WO has an Open or Pending Resolution Blocker | Blocker status, creator, timestamp, note | Navigates to Blocker Section in side panel |
| **Yellow** | Definition Change Flag | WO has one or more open Definition Change Flags | Change type, source entity, change description, who made the change | Navigates to Pending Definition Changes section in side panel |
| **White** | Supply Order Exception | WO has one or more associated Supply Order Lines with active Exceptions (`hasException = true`) | Exception note, who flagged it, when | Opens the Supply Order modal scoped to the affected line |

**Multiple indicators may appear simultaneously** on the same row (e.g., a
WO can have a Blocker AND a Definition Change Flag). Display order in row
layouts: Red → Yellow → White (most urgent first).

**Where indicators appear:**
- Operational lens grids: Purchasing, Receiving, Machining, Assembly,
  Distribution (subject to per-lens applicability — see below)
- Operations Lens grid
- Project View grid
- Detail Panel header when the panel is open on a row with the indicator

**Per-lens applicability of the White (Supply Order Exception) indicator:**
Not all lenses surface the White indicator. Operators in Machining,
Assembly, and Distribution Lens act on Blockers (Red Flag) when a Supply
Order Exception affects their work — the Exception itself is a procurement-
side signal that they don't act on directly. The White indicator appears in:
- Purchasing Lens (buyer follows up with vendor)
- Receiving Lens (receiver may have flagged it; visible to other receivers
  too)
- Operations Lens (manager visibility)
- Project View (manager visibility)

The Red Flag (Blocker) and Yellow Flag (Definition Change Flag) indicators
appear in all operational lens grids without exception.

**Hard rule:** Indicators are visual. Their hover content is informational
context. Their click actions navigate to the canonical surface for resolving
or reviewing the underlying record. The system never relies on indicators
alone for state — they are derived from the underlying record state and
update in real time as records change.

## Hard Rules Summary

A consolidated list of every hard rule in this document, for quick reference
during implementation and review.

| # | Rule |
|---|------|
| 1 | A Work Order is never called a batch. A Production Batch is never called a work order. |
| 2 | A batch dissolves if membership drops to one WO. |
| 3 | Batch priority = MAX priority of members. Batch due date = MIN due date of members. |
| 4 | Assembly routing templates may never include Purchase or Receive steps. |
| 5 | Assembly first step Ready only when all child WOs are Complete. |
| 6 | Assembly blockers apply only to the Assembly's own routing — not to child WOs. |
| 7 | `customerName` remains nullable. No constraint assumes every Project has a customer. |
| 8 | Template changes never cascade to open Work Orders (Principle 10). |
| 9 | Template edits require a confirmation screen showing affected Part count. |
| 10 | Distribution is always the final step in any routing sequence. |
| 11 | Scrap never satisfies demand. |
| 12 | CompletedQty falling below Demand creates a blocker. |
| 13 | Ready state is never set directly — always computed. |
| 14 | WO completion is transactional — final step and WO status update together. |
| 15 | Blocking a batched WO blocks the Batch, not the individual WO. |
| 16 | Duplicate children under the same BOM parent are blocked at DB and form level. |
| 17 | Circular BOM references are never allowed. Enforce via ancestry traversal. |
| 18 | Finished purchased components complete Distribution at Putaway, not in Distribution Lens. |
| 19 | Default Vendor changes on a Part flag WOs in Order Placed or later — no auto-update. |
| 20 | No blocker created without a note. No blocker resolved without a note. |
| 21 | Pending Resolution is not resolved. Work cannot proceed. |
| 22 | Demand Adjustment is not available when a parent Assembly WO depends on demand (Rev 2). |
| 23 | Demand may only be adjusted downward as a blocker resolution (Rev 2). |
| 24 | WO Split preserves parent Assembly demand relationships. |
| 25 | Every Blocker state change requires a note — no exceptions. |
| 26 | On any blocker resolution, user specifies outcome state — system does not auto-determine. |
| 27 | Pre-blocker state is always surfaced as suggested outcome state at resolution. |
| 28 | All member WOs in a Batch share the same state at all times. |
| 29 | Supply Order ETA is always editable — every change logged with user and timestamp. |
| 30 | One Supply Order = one expected receipt event. |
| 31 | Sub-status never drives state transitions. Blank sub-status is always valid. |
| 32 | Project Archival requires explicit manager confirmation. Not available to Operators. |
| 33 | Every state change records acting user and timestamp — no anonymous state changes. |
| 34 | Cancel is leaf-first only — must walk BOM tree leaf-first to cancel an Assembly with descendants. |
| 35 | Definition Change Flag resolution requires a note. Manager/Admin only. |
| 36 | Cancelled is single-valued at the WOStatus level — replaces prior status, not layered on top. |
| 37 | Stock Fulfillment Release is per-WO (Model 1), not per-Project. |
| 38 | Batching Confirm transitions WOStatus from Unreleased to Open — the second gate. |
| 39 | Skipped step state satisfies Ready prerequisite for downstream steps (alongside Complete). |
